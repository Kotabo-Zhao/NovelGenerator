二、功能一: 交互式大纲生成

2.1 设计目标

  • 用户提出修改意见后，系统解析为结构化指令（分类 + 影响范围 + 目标章节）
  • 最小化修改: 只重生成被反馈影响的章节，未提及部分保持不变
  • 反馈闭环: 用户意见必须确实生效，通过 diff 对比验证
  • 迭代追踪: 每次修改保存完整历史，支持回退

2.2 反馈分类体系

分类影响范围触发关键词处理策略
pacing (节奏)volume节奏、太快、太慢、高潮分布重生成受影响卷的章节数和节奏曲线
character_arc (角色弧线)global主角、成长、升级、反派、配角全量重生成大纲（嵌入角色弧线调整）
plot_logic (情节逻辑)volume矛盾、不合理、逻辑、漏洞重生成受影响卷（保持前后一致）
structure (结构)global卷、比例、三幕、太长、太短全量重生成（调整卷数和章数分布）
conflict (冲突)volume冲突、强度、不够刺激重生成受影响卷的冲突设计
hooks (钩子)chapter钩子、悬念、结尾、章末标记特定章节待重写
ending (结局)global结局、高潮、决战、收尾全量重生成（重点调整最后1-2卷）

关键设计决策: 用户反馈通过parse_feedback()从自由文本中提取7种结构化类别。global 级修改触发全量规划重生成(保留世界观和角色)，volume 级仅重生成受影响卷，chapter 级标记特定章节待重写。每次迭代保存完整历史，可通过 diff 验证变化确实生效。

三、功能二: 逻辑一致性校验 Agent

3.1 双层校验架构

层级方法速度覆盖误报率
L1 规则引擎 正则匹配 + 知识库查表 毫秒级 硬性约束（时空、姓氏、等级） 极低
L2 LLM 深度校验 DeepSeek API 语义推理 秒级 软性逻辑（情节合理性、角色决策） 低

3.2 检查维度

时空一致性 P0

  • 移动距离 vs 时间流逝（古代交通速度知识库）
  • 跨章位置跳跃检测（前章末尾 vs 本章开头）
  • 时间单位规范化（一炷香=0.5h，一息=5s）

角色关系一致性 P0

  • 直系血亲必须同姓规则
  • 辈分关系不可混淆
  • 年龄合理性（父辈≥子辈+15岁）

力量体系一致性 P1

  • 等级不可倒退检测
  • 能力边界不突变
  • 修为倒退需有原因（受伤/封印）

情节逻辑 P1

  • 事件因果链完整性
  • 角色决策符合性格设定
  • 道具/能力使用前后一致

3.3 集成点

Pipeline 集成: ConsistencyValidator 在章节生成完成后自动调用（generate_chapter_stream 末尾）。L1 规则引擎始终运行（无额外成本），L2 深度校验为可选（通过 run_deep 参数控制，减少 API 调用）。校验结果中的 P0 级违规触发自动重写流程。
违规严重度: P0 = 致命错误（-30分/条，必须修复），P1 = 明显问题（-15分/条，强烈建议修复），P2 = 轻微警告（-5分/条）。通过 build_fix_prompt() 将违规报告转化为 Writer 可执行的修复指令。

5.2 反转节奏规划

小说规模小反转中等反转大反转
<100章5个2个1个
100-300章12个4个2个
>300章20个8个3个

5.3 反转适用性判断

# 判断某章是否适合放置反转
✅ 适合: 卷末章、每5章周期点、非高潮过渡章
❌ 不适合: 已有强度5冲突的章节、卷首章（需先建立预期）
🎯 最佳: 幕转折处（信息炸弹）、卷末（身份揭露）

# 反转后处理
- 反转前保持自然叙述，不刻意暗示
- 反转点放在 70%-90% 位置（给读者消化空间）  
- 反转后留白——不立即解释，让冲击力沉淀
- 确保前文至少有一个可回溯的微小线索

6.4 关键设计决策

1. 双层校验架构 (ConsistencyValidator)

L1 规则引擎（毫秒级正则匹配）覆盖硬性约束，L2 LLM（秒级语义推理）覆盖软性逻辑。L1始终运行（零API成本），L2可选（通过 run_deep 参数控制）。这种设计在成本和覆盖之间取得平衡——大部分常规错误被L1捕获，复杂逻辑问题由L2处理。

2. 反馈→指令转化 (OutlineInteractive)

用户的自然语言反馈被解析为7种结构化类别，每种有确定的影响范围（global/volume/chapter）。这个转化是关键——确保"修改意见确实生效"。global级触发全量重生成（成本高但影响大），chapter级仅标记待重写（成本低）。每次迭代保存完整历史+diff输出，用户可验证变化。

3. 分析-修复分离模式

ConsistencyValidator、OpeningOptimizer、TwistDesigner 都遵循"分析 → 生成修复提示 → Writer执行修复"的分离模式。校验Agent不直接修改文本，而是输出结构化修复指令（build_fix_prompt()），由Writer在重写时注入上下文中。这保持了Writer的对文本质量的最终控制权。

4. 反转与伏笔联动

TwistDesigner 规划的反转点会自动向 ForeshadowingDesigner 注册伏笔需求。每个反转类型有明确的所需伏笔数量（身份揭露≥3个，信息炸弹≥4个）。反转章节生成时，Memory的L3层自动注入相关伏笔提醒，确保Writer埋设足够线索。
