二、Agent 记忆方案参考

2.1 Soul File 模式（业界参考）

"Soul 文件"是 Agent 社区的一种实践模式：将 Agent 的核心身份、知识、状态持久化为一个结构化文件，Agent 每次启动时加载，运行中增量更新。

标准 Soul 文件结构

{
  "identity": {        // 身份: 我是谁
    "name": "...",
    "role": "小说创作引擎",
    "capabilities": [...]
  },
  "knowledge": {       // 知识: 我知道什么
    "rules": [...],    // 规则/红线
    "patterns": [...]  // 经验模式
  },
  "context": {         // 上下文: 当前在做什么
    "current_task": "...",
    "progress": {...}
  },
  "memory": {          // 记忆: 历史经验
    "recent": [...],   // 近期
    "archived": [...]  // 压缩归档
  },
  "meta": {            // 元数据
    "version": 1,
    "updated": "..."
  }
}

与 NovelGenerator 的映射

Soul 字段→NovelGenerator 文件
identity→plan.json (worldbuilding + characters)
knowledge→styles.py + humanizer.py (规则引擎)
context→state.json + global_state.json
memory→chapters/*.md + foreshadowing.json
meta→plan.json._meta
关键发现：当前 NovelGenerator 的 6 个分散文件已经天然构成了一个"Soul 文件"体系，只是缺乏统一的访问层。改造的核心不是"新建一个 Soul 文件"，而是"为已有的 Soul 体系加一层统一管理器"。

三、五维可行性评估

3.1 技术可行性 · 82/100

✅ 已有基础

  • atomic_io.py 提供了原子写入 + 损坏恢复，是文件级持久化的可靠底层
  • NovelMemory 已有 L1/L2/L3 分层上下文构建，只是耦合在 Engine 内
  • 所有模块通过 Engine 串行调用，天然避免了真正的并发竞争

⚠️ 技术障碍

障碍严重度解决方案
LLM 上下文窗口限制（~8K tokens） 高 渐进式摘要压缩：每10章自动生成压缩摘要，旧章仅保留摘要而非全文。RAG 检索：用嵌入向量索引"记忆片段"，按需注入而非全量注入。
记忆膨胀：1000章 ≈ 5MB 文本 中 文件存储本身不是问题（磁盘廉价）。瓶颈在 LLM token 预算。压缩策略同上。
多类型记忆的结构化存储 低 已有 6 种文件格式覆盖了当前需求。新增记忆类型只需新增 JSON 文件或扩展已有结构，无技术障碍。
跨小说记忆共享 中 当前每本小说独立目录。如需"从上一本小说学习写作偏好"，需增加全局用户偏好文件（~/.novelgen/prefs.json）。

3.2 架构兼容性 · 85/100

当前架构的优势

  • Engine 是唯一入口：所有模块通过 Engine 调用，天然适合插入一个中间层
  • 文件系统为存储：不需要数据库迁移，不需要 ORM，集成成本极低
  • 已有分层概念：L1/L2/L3 的分层记忆已经建立了"核心→近期→远期"的模型

集成方案：SharedMemoryManager

# 新增: backend/core/shared_memory.py
class SharedMemoryManager:
    """统一记忆访问层 — 所有模块通过此接口读写记忆"""
    
    def get_novel_soul(novel_id) → SoulSnapshot
    def update_soul(novel_id, changes) → bool
    def get_context_for_module(module, novel_id, chapter) → str
    def append_memory(novel_id, memory_type, data)
    def query_memory(novel_id, query, top_k=5) → list
    
    # 内部:
    # - 管理 6 种文件的统一读写
    # - 维护内存缓存（带 TTL 自动失效）
    # - 提供变化通知（callback 机制）
    # - 渐进式摘要压缩调度
集成难度评估：新增一个模块（~400行），修改 Engine.__init__ 注入 SharedMemoryManager，各模块将 self.memory.xxx() 调用替换为 self.shared_memory.xxx()。无需修改模块内部逻辑，约 2-3 小时工作量。

3.3 性能影响 · 68/100

这是五维中最关键的瓶颈。核心矛盾：LLM 的上下文窗口是有限且昂贵的，但长篇小说会产生海量文本。

量化分析

场景章节数正文量全量注入L1/L2/L3注入摘要压缩注入
短篇30章~90K字45K tokens ❌3K tokens ✅3K tokens ✅
中篇100章~300K字150K tokens ❌4K tokens ✅5K tokens ✅
长篇500章~1.5M字750K tokens ❌8K tokens ⚠️6K tokens ✅
超长篇1000章~3M字1.5M tokens ❌15K tokens ❌8K tokens ✅

性能策略

策略1: 渐进式摘要压缩

每写完10章，用一个轻量 LLM 调用将前10章压缩为300字摘要。旧章正文不再注入 prompt，只注入压缩摘要 + 最近3章全文。

效果: 1000章的记忆从 1.5M tokens → ~8K tokens

策略2: RAG 语义检索

对章节摘要做嵌入(embedding)，当需要特定信息时（如"主角的剑叫什么"），检索最相关的记忆片段注入。

代价: 嵌入生成 + 向量检索增加 ~1s延迟

策略3: 分层缓存

内存缓存最近访问的记忆对象（TTL 30s），避免每次 API 调用都重新读取文件和构建上下文。

效果: 减少 60-80% 的磁盘 I/O

策略4: 记忆优先级

不是所有记忆都平等。L1（核心设定）永远注入，L2（近期上下文）仅3章，L3（伏笔）按紧迫度排序取 Top 8。

已实现: 当前 NovelMemory 已有此分层

推荐方案 A（乐观锁）：每个 JSON 文件增加 "_version": N 字段。写入时检查版本号，如果不匹配则重试（读取最新→合并→写入）。由于当前系统所有写操作都通过 Engine，最多 3 次重试即可解决 99.9% 的竞态。

乐观锁实现示例

# shared_memory.py
def atomic_update(novel_id, file_name, updater_fn, max_retries=3):
    """乐观锁写入: 读取→修改→版本检查→写入"""
    for attempt in range(max_retries):
        data = safe_read_json(path)
        version = data.get("_version", 0)
        
        new_data = updater_fn(data)  # 用户提供的修改函数
        new_data["_version"] = version + 1
        
        # 原子写入 + 版本检查（通过临时文件实现）
        if _try_write_with_version(path, new_data, version):
            return True  # 成功
        # 版本冲突 → 重试
    raise ConcurrencyError(f"Failed after {max_retries} retries")

3.5 维护成本 · 80/100

✅ 降低维护成本的方面

  • 统一记忆层消除各模块各自维护文件引用的代码重复
  • 中央化的记忆格式变更只需改一处（迁移脚本）
  • 缓存层减少磁盘 I/O，降低"文件损坏→全局崩溃"的风险面
  • 新 Agent 加入时无需关心文件路径，调用 shared_memory.get_xxx() 即可

⚠️ 增加维护成本的方面

  • Schema 迁移：记忆格式升级时需要迁移已有小说数据
  • 新增一个模块意味着多一个故障点
  • 团队成员需要理解统一记忆模型
  • 调试时需要在 SharedMemoryManager 层加日志

六、最终结论

可行
建议立即启动 Phase 1，Phase 2 在长篇小说（>100章）场景触发

总工作量估计：Phase 1 ≈ 2天，Phase 2 ≈ 4天

核心理由

  1. 现有基础好：6 个文件已构成事实上的 Soul 文件体系，改造是"加一层"而非"推翻重来"
  2. 架构友好：Engine 的单入口设计使集成成本极低（~20 行改动）
  3. 性能可控：渐进式摘要压缩 + 分层记忆已解决核心瓶颈
  4. 风险可管理：乐观锁 + 原子 I/O 保证了数据安全；自动备份 + 向后兼容保证了迁移安全
  5. 收益显著：统一记忆层消除代码重复，新 Agent 即插即用，长期维护成本 < 当前方案

不做会怎样？

  • 每新增一个 Agent，就要在 Engine 中手动传递 3-5 个文件路径/数据对象 → 代码耦合越来越深
  • 长篇小说（>300 章）的记忆膨胀会导致 LLM 上下文溢出 → 生成质量断崖式下降
  • 多个 Agent 独立维护各自的状态快照 → 数据不一致风险随 Agent 数量指数增长
